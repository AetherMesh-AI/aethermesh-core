# Model and Expert Manifest Versions 1 through 3

A model or expert stored locally may carry an adjacent `*.manifest.json` file. This is a small, hand-authored audit document, not a registry, network advertisement, capability guarantee, consensus proof, or reward record.

`aethermesh_core.expert_manifest.load_expert_manifest()` validates the exact JSON version 1, 2, or 3 shape. Version 2 adds required manifest identity and capability metadata; version 3 adds the required output contract reference. The reader continues to accept existing version 1 and 2 records unchanged. `expert_is_usable()` is deliberately stricter: it returns true only when validation is `passed`, the adjacent artifact exists and matches its SHA-256 hash, and the recorded local validation receipt preserves the manifest's accepted local identifier(s), schema references, lineage, contribution attribution, artifact hash, validation time, and validator. A `receipt_path` must be `null` until a real local validation creates it; the sample remains `unvalidated` and does not invent one.

## Required top-level fields

| Field | Meaning |
| --- | --- |
| `version` | Exactly `1` for the original shape, `2` for capability metadata, or `3` for manifests with an explicit output contract. This is the local manifest schema/record-format version, not a model or expert capability version. |
| `manifest_id` | Required stable local manifest identifier. Together with `version`, `creator_node_id`, `lineage.parent_manifest_ids`, and `contribution_attribution`, it remains unchanged when capability metadata is loaded, copied, validated, or read for routing. |
| `model_id` or `expert_id` | At least one stable, non-empty, whitespace-free local identifier. Both identifiers are permitted when a model is packaged as an expert. Use a clear local ID such as `local-echo-fixture-v0`, and retain it across local validation runs unless the model/expert meaningfully changes. These IDs are not global registry identifiers or capability claims. |
| `name` | Required non-empty, local-first human-readable descriptive metadata for logs, validation output, and manifest summaries, such as `Local Echo Fixture Expert`. It is not a primary lookup key or an immutable identifier; use `model_id` or `expert_id` for lookup and provenance. |
| `creator_node_id`, `created_at` | Originating local node and immutable UTC creation timestamp in `YYYY-MM-DDTHH:MM:SSZ` form. Populate `creator_node_id` from the stable local node identity, not a placeholder. `created_at` is when this manifest was first authored locally, not model training time, network acceptance time, validation time, contribution time, or reward eligibility. Preserve both during copies and validation; later activity belongs in validation receipts or future update records. Fixtures may supply a fixed UTC value for reproducible local tests. |
| `author`, `owner`, `attribution_notes` | Required human-readable provenance. `author` identifies who authored this entry; `owner` identifies the local operator accountable for it. They are deliberately separate from immutable `creator_node_id`, so an accountable operator does not overwrite creator lineage. `attribution_notes` records local provenance context. This metadata is for provenance, validation, and contribution tracking only—not rewards, tokenomics, rankings, governance, network authority, or control over other nodes. Preserve all three during a copy or a non-attribution update. An ownership change must be an explicit successor record: retain `creator_node_id`, record the prior manifest in `lineage.parent_manifest_ids`, and state the transfer in `attribution_notes`. |
| `artifact_hash` | Required stable artifact identity. For a concrete local artifact it exactly matches `artifact.sha256`. For an early non-model expert with no concrete artifact, it is a clearly marked `placeholder:sha256:` hash derived from canonical JSON containing only `version`, `expert_id`/`model_id`, `creator_node_id`, and the declared relative artifact reference or interface name. The placeholder deliberately excludes timestamps, machine-local paths, random values, and environment data; it is not a content hash. |
| `artifact` | Adjacent/local relative `reference` plus lowercase `sha256:` content hash when a concrete artifact exists. Set `sha256` to `null` only for the deterministic non-model placeholder case. |
| `input_schema_ref` | Version 2 and 3: a safe local relative reference to a valid JSON Schema draft 2020-12 object contract with required, typed, constrained inputs. Loading never resolves this contract over the network. |
| `output_schema_ref` | Version 3 only: a safe local relative reference to the exact JSON Schema draft 2020-12 output contract the expert declares. It must name a readable schema with one JSON output type; an informal label is not a reference. Nested `$ref` and `$dynamicRef` values must be local fragments so output validation never retrieves another schema over the network. |
| `supported_task_categories` | Non-empty, bounded labels for tasks the operator intends to support. |
| `runtime_requirements` | Non-empty local requirements such as a runner or model format. |
| `capabilities` | Required non-empty capability entries. Each entry has a stable `capability_id`, human-readable `name` and short `description`, `input_modality`, `output_modality`, `supported_task_type`, non-marketing local constraints, and known limitations. Its nested validation is either explicitly `unvalidated` with no evidence fields, or `passed`/`failed` with a local test name, validation receipt ID, UTC validation time, and result summary. Capability IDs are unique within the manifest. |
| `lineage` | Source model, optional adapter/template references, local-change note, parent IDs, and derived artifact references. |
| `validation` | Test command, expected input reference, receipt path, timestamp, status, and validator node ID. All evidence fields are required for `passed`; they may be `null` before validation. |
| `contribution_attribution` | Creator (matching the top level), modifiers, validator, and receipt references. It records attribution only—never rewards, credits, or token accounting. |

All references are safe relative paths, and usability checks reject references that resolve outside the manifest directory. During a local prototype validation, the validator resolves `output_schema_ref` from the manifest directory, reads the declared JSON Schema without a network request, and validates the produced output against that schema before writing its receipt. Validation and contribution attribution must name the same validator and receipt. The format rejects unknown fields so a handoff or copy keeps the same complete provenance shape. `examples/model-experts/echo-expert-v0/manifest.json` sits next to its artifact and is a parseable, deliberately unvalidated root example.

An output-schema change is a contract change: create a new version 3 manifest with a new `manifest_id` and record the prior manifest in `lineage.parent_manifest_ids`; do not silently retarget `output_schema_ref` in an already validated manifest. This keeps the old receipt bound to its manifest, schema reference, creator, and attribution while local validators can audit the new contract separately.

## Local routing reads

Local routing may read `capabilities` to compare a requested task type and input/output modality with the declared local constraints and validation state. It must treat `unvalidated`, failed, or stale evidence conservatively and must not infer performance, autonomy, production readiness, network availability, or consensus from a capability label. Routing is read-only: it must not rewrite `manifest_id`, `version`, `creator_node_id`, `lineage`, `contribution_attribution`, or capability validation evidence. A later validation or manifest revision is a separately authored record, not a routing-side mutation.

Version 1 contains the fields above except `manifest_id`, `capabilities`, `input_schema_ref`, `output_schema_ref`, `author`, `owner`, and `attribution_notes`; version 2 additionally includes `manifest_id`, `capabilities`, and `input_schema_ref`; version 3 requires every listed field. Existing version 1 and 2 manifests remain readable and retain the version they were created with. Their creator node ID, lineage, receipt links, and contribution attribution must not be rewritten merely to adopt a later format.

A passed manifest's receipt is an exact JSON object with `receipt_version` set to `aethermesh-expert-validation-receipt/v0`, `status` set to `passed`, and matching `name`, `manifest_id` and `input_schema_ref` for version 2 or later, `output_schema_ref`, `author`, `owner`, `attribution_notes`, `lineage`, and `contribution_attribution` for version 3, plus `model_id` and/or `expert_id`, `creator_node_id`, `created_at`, `artifact_hash`, `validated_at`, and `validator_node_id` values. The receipt references immutable creation and attribution data; it does not replace or regenerate them. The receipt displays the descriptive name alongside stable IDs; it never treats the name as a primary lookup key. A placeholder identity records provenance but cannot make an expert usable until a concrete artifact and matching content hash exist. Merely creating a file at `receipt_path` does not make an expert usable.
